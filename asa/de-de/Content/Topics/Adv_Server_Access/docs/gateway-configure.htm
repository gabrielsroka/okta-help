Konfiguration des Advanced Server Access -Gateways

In diesem Thema wird erklärt, wie Sie ein Advanced Server Access-Gateway konfigurieren.

Befehlszeilenoptionen

    service: Führt den Dienst sft-gatewayd aus.

    support: Erfasst lokale Systemdaten für den Okta-Support.

    -h, --help: Zeigt die Hilfe an.

    -v, --version: Zeigt die Version an.

    --syslog: Erzwingt die Syslog-Protokollierung

Konfigurationsdatei

Sie können Advanced Server Access -Gateways mit einer Konfigurationsdatei steuern, die Sie unter /etc/sft/sft-gatewayd.yaml finden. Nach der Installation des Gateways ist eine Muster-Konfigurationsdatei unter /etc/sft/sft-gatewayd.sample.yaml verfügbar. Wenn die Konfigurationsdatei nicht erstellt wurde oder nicht verfügbar ist, verwendet das Gateway die folgenden Standardwerte.

Sie müssen das Gateway neu starten, bevor Änderungen an der Konfigurationsdatei wirksam werden. Detaillierte Informationen finden Sie unter Verwaltung des Advanced Server Access-Gateways.

Einrichtungstoken-Optionen

Diese Optionen steuern, wie Sie ein Einrichtungstoken zum Gateway hinzufügen. Sie müssen eine der folgenden Optionen aktivieren. Detaillierte Informationen zum Abrufen eines Einrichtungstokens finden Sie unter Erstellen eines Einrichtungstokens für ein Advanced Server Access-Gateway.

Option Standardwert Beschreibung
SetupToken nicht festgelegt Gibt ein Einrichtungstoken an, das auf der Advanced Server Access-Plattform erstellt wurde.

Hinweis: Wenn Sie diese Option verwenden, bleibt das Einrichtungstoken im Klartext verfügbar. Okta empfiehlt, die Leserechte auf die Konfigurationsdatei einzuschränken (z. B. 0600 unter Linux).
SetupTokenFile Linux: /var/lib/sft-gatewayd/setup.token

FreeBSD: /var/db/sft-gatewayd/setup.token

 Gibt den Pfad zu einer separaten Datei an, die das Einrichtungstoken enthält. Der Standardwert hängt vom Betriebssystem des Gateways ab.

Diese Option ist die empfohlene Methode.

Wenn Sie diese Option verwenden, müssen Sie die Datei für das Einrichtungstoken manuell erstellen und das auf der Advanced Server Access-Plattform erstellte Einrichtungstoken zur Datei hinzufügen.

Wenn das Gateway bei einem Team registriert ist, löscht das Gateway die Token-Datei.

Hinweis: Wenn die Option SetupToken ebenfalls konfiguriert ist, verwendet das Gateway das dort festgelegte Token.

Protokolloptionen

Option Standardwert Beschreibung
LogLevel info Steuert die Ausführlichkeit der Protokolle. Okta empfiehlt, den Wert info beizubehalten.

Zu gültigen Werten gehören:

  • error
  • warn
  • info
  • debug

Verbindungsoptionen

Option Standardwert Beschreibung
AccessAddress 1.1.1.1 Gibt die Netzwerkadresse (IPv4 oder IPv6) an, die Clients für den Zugriff auf das Gateway verwenden. Wenn keine Adresse festgelegt ist, verwendet das Gateway die Adresse, die von der Netzwerkschnittstelle oder den Metadaten des Cloud-Anbieters vorgegeben werden.
AccessPort 7234 Gibt den Port an, über den Clients auf das Gateway zugreifen können.
ListenAddress 0.0.0.0 Gibt die Netzwerkadresse (IPv4 oder IPv6) an, die Clients zum Überwachen auf Verbindungen verwenden. Standardmäßig überwacht das Gateway jede verfügbare Schnittstelle auf Verbindungen.
ListenPort 7234 Gibt den Port an, die Clients zum Überwachen auf Verbindungen verwenden.
TLSUseBundledCAs True Zwingt das Gateway dazu, für die Sicherung von HTTP-Anforderungen mit TLS den gebündelten Zertifikatspeicher zu verwenden anstatt den Zertifikatspeicher des Betriebssystems. Das gilt auch für Anforderungen an den Advanced Server Access-Cloud-Dienst.

Hinweis: Um den Zertifikatspeicher des Betriebssystems zu verwenden, legen Sie für diese Option „false“ fest.
RefuseConnections False Steuert, ob das Gateway SSH- und RDP-Proxy-Datenverkehr akzeptiert. Wenn diese Option aktiviert ist, werden SSH- und RDP-Verbindungsanforderungen nicht geroutet und Proxy-Datenverkehrsanforderungen werden vom Gateway nicht überwacht.
ForwardProxy nicht festgelegt Gibt die URL eines HTTP CONNECT-Proxys an, der für ausgehende Netzwerkverbindungen zu Advanced Server Access genutzt wird. Alternativ können Sie den Proxy auch mit der Umgebungsvariablen HTTPS_PROXY festlegen.

LDAP-Optionen

Diese Optionen steuern, wie das Gateway sichere Verbindungen zu LDAP-Servern herstellt. Sie müssen diese Optionen konfigurieren, damit das Gateway mit der AD-Einbindung funktioniert. Siehe Konfiguration eines Advanced Server Access-Gateways für die AD-Einbindung.

Standardmäßig startet das Gateway TLS, führt aber keine Zertifikatvalidierung durch. Das Gateway unterstützt nur StartTLS für die LDAP-Kommunikation. LDAPS wird von Gateways nicht unterstützt.

Sie müssen diese Optionen in einem LDAP-YAML-Wörterbuch mit zwei Leerzeichen einrücken:

LDAP:
  StartTLS: ...

Option Standardwert Beschreibung
StartTLS True

Führt ein Upgrade der Active Directory-/LDAP-Serverkommunikationen auf TLS durch. Wenn diese Option deaktiviert ist, wird die Kommunikation mit dem Server nicht verschlüsselt.
ValidateCertificates False Führt die Validierung der Zertifikate durch, die der AD- bzw. LDAP-Server übermittelt. Wenn diese Option deaktiviert ist, sind die Serverkommunikationen möglicherweise anfällig für MITM-Angriffe.

Wenn diese Option aktiviert ist, sucht das Gateway unter dem Pfad, der von TrustedCAsDir vorgegeben wird, nach Zertifikaten.

Teams sollten sicherstellen, dass gültige Zertifikate für alle verbundenen Domänen auf dem Gateway gespeichert sind, bevor sie diese Option aktivieren. Werden keine gültigen Zertifikate gefunden, verweigert das Gateway alle Verbindungen.

TrustedCAsDir nicht festgelegt Gibt den Pfad zu einem Verzeichnis an, das öffentliche Zertifikate einer Zertifizierungsstelle enthält (für gewöhnlich in Active Directory verwaltet und über eine Gruppenrichtlinie verteilt). Das Verzeichnis und die Zertifikate müssen für den sft-gatewayd-Benutzer lesbar sein, und die Zertifikate müssen PEM-codiert sein. Unterverzeichnisse werden nicht überprüft.

Sie können jeden beliebigen Pfad verwenden. Okta empfiehlt jedoch die Verwendung von /etc/sft/trusted-ldap-certs/.

Informationen zu Zertifikaten finden Sie unter Hinzufügen eines Zertifikats zu einem Advanced Server Access-Gateway.

RDP-Optionen

Diese Optionen steuern, wie das Gateway Remotedesktopprotokoll (RDP)-Sitzungen verwaltet. Teams müssen diese Optionen so konfigurieren, dass RDP-Sitzungen über die Gateways zu den Servern geroutet werden, die beim Server-Agenten registriert sind oder in einer AD-Domäne ermittelt wurden. Sie müssen diese Optionen konfigurieren, damit das Gateway mit der AD-Einbindung funktioniert. Siehe Konfiguration eines Advanced Server Access-Gateways für die AD-Einbindung.

Sie müssen RDP-Verbindungen mit den Optionen Enabled sowie entweder TrustedCAsDir oder DangerouslyIgnoreServerCertificates explizit zulassen.

Wenn ein Team die Funktion zur AD-Einbindung verwendet, kann das Gateway auch dann noch AD-Server erkennen, wenn die RDP-Unterstützung nicht aktiviert ist. Siehe Active Directory-Serverermittlung.

Sie müssen diese Optionen in einem RDP-YAML-Wörterbuch mit zwei Leerzeichen einrücken:

RDP:
  Enabled: ...

Option Standardwert Beschreibung
Enabled False

Steuert die RDP-Funktionalität für das Gateway. Diese Option ist standardmäßig deaktiviert, da RDP eine zusätzliche Konfiguration erfordert. Muss auf True festgelegt werden, bevor RDP-Verbindungen zugelassen werden.

Teams können RDP-Verbindungen nur über Gateways routen, auf denen Ubuntu 20.04 ausgeführt wird.
TrustedCAsDir nicht festgelegt Gibt den Pfad zu einem Verzeichnis an, das öffentliche Zertifikate enthält, die von einer Zertifizierungsstelle signiert sind (für gewöhnlich in Active Directory verwaltet und über eine Gruppenrichtlinie verteilt). Das Verzeichnis und die Zertifikate müssen für den sft-gatewayd-Benutzer lesbar sein, und die Zertifikate müssen PEM-codiert sein. Unterverzeichnisse werden nicht überprüft.

Sie können jeden beliebigen Pfad verwenden. Okta empfiehlt jedoch die Verwendung von /etc/sft/trusted-rdp-certs/.

Informationen zu Zertifikaten finden Sie unter Hinzufügen eines Zertifikats zu einem Advanced Server Access-Gateway.
DangerouslyIgnoreServerCertificates False Verhindert, dass das Gateway bei der Herstellung einer Verbindung zu einem RDP-Host Serverzertifikate validiert. Dieses Flag ist außerhalb von Testumgebungen riskant, kann aber erforderlich sein, wenn der RDP-Host über selbstsignierte Zertifikate verfügt.
MaximumActiveSessions 20 Steuert die Anzahl an gleichzeitigen RDP-Sitzungen, die das Gateway zulässt. Wenn diese Anzahl an Sitzungen erreicht wurde, sehen alle weiteren Benutzer eine Fehlermeldungen und können keine Verbindung zum Gateway herstellen. Diese Option wird ausschließlich aus ressourcen- und leistungsbezogenen Gründen verwendet.
VerboseLogging True Steuert den Protokolliergrad von RDP-internen Protokollen. Diese Nachrichten sind bei der Problemdiagnose hilfreich, machen Protokolle unter Umständen auch unübersichtlich.

Hinweis: Legen Sie hier False fest, um alle internen RDP-Protokollnachrichten als „debug“ zu kennzeichnen.

Optionen für die Sitzungserfassung

Option Standardwert Beschreibung
SessionLogFlushInterval 10s Gibt einen Größen-Schwellenwert für Sitzungserfassungsprotokolle an. Wenn dieser Schwellenwert erreicht wurde, werden Protokolle für eine aktive Sitzung signiert, auf den Datenträger gespeichert und aus dem Speicher gelöscht. Diese Option muss eine Zeiteinheit enthalten (ms, s, m, h).
SessionLogMaxBufferSize 262144 Gibt einen Größen-Schwellenwert für Sitzungserfassungsprotokolle an. Wenn dieser Schwellenwert erreicht wurde, werden Protokolle für eine aktive Sitzung signiert, gespeichert und aus dem Speicher gelöscht Diese Option wird in Bytes gemessen.
SessionLogTempStorageDirectory /tmp Gibt ein temporäres Verzeichnis an, in dem SSH-Sitzungsprotokolle vor dem Hochladen zum speziellen Protokollziel gespeichert werden.

Der Standard hängt vom Betriebssystem ab. Unter Linux ist der Standard im Allgemeinen /tmp, sofern die Umgebungsvariable $TMPDIR nicht konfiguriert ist.
LogDestinations nicht festgelegt Gibt an, wo die fertigen Sitzungsprotokolle gespeichert werden. Teams können Protokolle lokal oder in AWS- oder GCS-Buckets speichern. Sie können Sitzungsprotokolle an mehreren Speicherorten speichern. In diesem Fall werden die Protokolle in der Reihenfolge an das Ziel gesendet, die in der Konfigurationsdatei angegeben ist. Sitzungsprotokolle werden nie an Advanced Server Access gesendet.

Nachfolgend finden Sie die Standardwerte und Beispiele.

Beispiel: Linux/BSD

LogDestinations:
  - Type: file
    LogDir: /var/log/sft/sessions

Gateways können nicht in die Verzeichnisse /home, /root oder /run/user schreiben.

Beispiel: AWS

LogDestinations:
  - Type: s3
    Bucket: BUCKET-NAME
    # Region ist optional
    Region: US-EAST-1
    # Sie können ein Profil für gemeinsam genutzte Anmeldedaten angeben ...
    Profile: AWS-PROFILE-NAME
    # ... oder AccessKeyId, SecretKey und SessionToken für statische Anmeldedaten ...
    AccessKeyId: SECRET
    SecretKey: SECRET
    SessionToken: SECRET
    # ... oder keine Anmeldedaten, um die standardmäßige Anmeldedatenkette zu verwenden

Beispiel: Google Cloud

LogDestinations:
  - Type: gcs
    Bucket: BUCKET-NAME
    # Geben Sie entweder CredentialsFile oder CredentialsJSON an (oder nichts, um die Anmeldedaten der Instanz zu verwenden)
    CredentialsFile: /path/to/cred/file
    CredentialsJSON: |
    {
    "SOME": "value",
    "IN": "json"
    }

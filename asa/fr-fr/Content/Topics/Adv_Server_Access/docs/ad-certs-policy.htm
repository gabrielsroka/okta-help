Configurer les stratégies de groupe pour les serveurs AD

Une fois que les équipes ont importé un certificat dans Active Directory (AD), elles doivent encore configurer les ressources du serveur à utiliser avec Advanced Server Access. Okta recommande aux équipes de placer tous les serveurs protégés par Advanced Server Access dans une unité d'organisation (OU) distincte des autres ressources du domaine. Les équipes peuvent gérer les contrôleurs de domaine et les ressources au sein de l'OU de manière centralisée en utilisant un objet de stratégie de groupe (GPO).

Avertissement : les équipes ne doivent pas mettre en place l'authentification par carte à puce pour accéder aux contrôleurs de domaine tant que l'authentification sans mot de passe n'a pas fait l'objet de tests approfondis. Dans le cas contraire, les équipes risquent d'être exclues du domaine.

Avant de commencer

  • Créer un certificat autosigné
  • Créer un certificat avec une demande de signature de certificat

Commencer la tâche

  1. Distribuez le certificat en utilisant un GPO.
    1. Faites un clic droit sur l'objet des contrôleurs de domaine et sélectionnez Create a GPO in this domain, and Link it here… (Créer un GPO dans ce domaine et le lier ici…).
    2. Dans la fenêtre New GPO (Nouveau GPO), nommez le GPO Advanced Server Access - Certificate.
    3. Cliquez sur OK.
    4. Dans l'arborescence de la console, ouvrez Configuration de l'ordinateur\Stratégies\Paramètres Windows\Paramètres de sécurité\Stratégies de clé publique.
    5. Faites un clic droit sur Trusted Root Certification Authorities (Autorités de certification racines de confiance) et cliquez sur Importer.
    6. Dans Welcome to the Certificate Import Wizard (Bienvenue dans l'Assistant Importation du certificat), cliquez sur Suivant.
    7. Indiquez le chemin d'accès au fichier de certificat que vous avez précédemment créé et cliquez sur Suivant.
    8. Cliquez sur Place all certificates in the following store (Placer tous les certificats dans la base suivante) et cliquez sur Suivant.
    9. Vérifiez les informations et cliquez sur Terminer.
  2. Désactivez l'authentification au niveau du réseau.
    1. Depuis l'écran Group Policy Management (Gestion des stratégies de groupe), identifiez le GPO Advanced Server Access - Certificate.
    2. Faites un clic droit sur le GPO et cliquez sur Modifier.
    3. Dans l'arborescence de la console, ouvrez Configuration de l'ordinateur\Stratégies\Modèles d'administration\Composants Windows\Services Bureau à distance\Hôte de session Bureau à distance\Sécurité.
    4. Double-cliquez sur Require user authentication for remove connections by using Network Level Authentication (Requérir l'authentification utilisateur pour les connexions à distance à l'aide de l'authentification au niveau du réseau).
    5. Dans la fenêtre des propriétés, sélectionnez Désactivé(e).
    6. Cliquez sur Appliquer, puis sur OK.
  3. Affectez le GPO à l'OU Advanced Server Access.
    1. Identifiez l'OU Advanced Server Access à partir du composant logiciel enfichable des stratégies de groupe.
    2. Faites un clic droit sur le contrôleur de domaine, puis cliquez sur Link an Existing GPO (Lier un objet de stratégie de groupe existant).
    3. Dans la fenêtre Select GPO (Sélectionner un GPO), sélectionnez le GPO Advanced Server Access - Certificate.
    4. Cliquez sur OK.
  4. Exigez l'authentification par carte à puce pour les comptes.
    • N'appliquez pas cette stratégie aux contrôleurs de domaine tant que l'authentification sans mot de passe n'a pas fait l'objet de tests approfondis. Dans le cas contraire, vous risquez d'être exclus du domaine.
    • Si votre organisation Okta utilise l'authentification déléguée Active Directory, l'application de cette stratégie aux comptes utilisateurs empêchera ces derniers d'accéder au tableau de bord Okta.
    1. Faites un clic droit sur l'OU Advanced Server Access et sélectionnez Create a GPO in this domain, and Link it here… (Créer un GPO dans ce domaine et le lier ici…).
    2. Dans la fenêtre New GPO (Nouveau GPO), nommez le GPO Advanced Server Access - Authentication.
    3. Cliquez sur OK.
    4. Dans l'arborescence de la console, ouvrez Configuration de l'ordinateur\Stratégies\Paramètres Windows\Paramètres de sécurité\Stratégies locales\Options de sécurité.
    5. Double-cliquez sur Interactive logon: Require Windows Hello for Business or smart card (Ouverture de session interactive : Windows Hello Entreprise ou carte à puce nécessaire).
      Remarque : certaines versions de Windows utilisent plutôt Interactive logon: Require smart card (Ouverture de session interactive : carte à puce nécessaire).
    6. Dans la fenêtre des propriétés, sélectionnez Définir ce paramètre de stratégie.
    7. Sélectionnez Activé(e).
      Avertissement : les équipes ne doivent pas mettre en place l'authentification par carte à puce pour accéder aux contrôleurs de domaine tant que l'authentification sans mot de passe n'a pas fait l'objet de tests approfondis. Dans le cas contraire, les équipes risquent d'être exclues du domaine.
    8. Cliquez sur Appliquer, puis sur OK.

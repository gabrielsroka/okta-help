Composants Advanced Server Access

Le déploiement de l'Advanced Server Access comprend l'ensemble des composants suivants :

Composants Description
Équipes Une équipe est un conteneur principal. Il contient toutes les ressources nécessaires à un déploiement déterminé. Chaque équipe dispose d'un nom unique et d'un fournisseur d'identité (IdP).

La portée de tous les autres objets de configuration de l'Advanced Server Access s'étend à une équipe spécifique.
Groupes Un groupe est un ensemble d'utilisateurs qui dispose d'un ensemble de permissions associées. Lors de chaque déploiement, deux groupes sont créés par défaut : tout le monde et propriétaires.

Il est possible d'assigner plusieurs rôles d'équipe à un groupe. Chaque membre d'un groupe hérite des rôles assignés.
Projets Un projet est une étendue d'autorisation. Les projets sont semblables aux domaines de l'Active Directory. Chaque projet associe un ensemble de ressources (telles que des utilisateurs et des serveurs) à un jeu de configurations, qui comprend le contrôle d'accès basé sur les rôles (RBAC) et les règles de contrôle d'accès.
Identifiants dynamiques Les identifiants Advanced Access Server sont des objets éphémères utilisés pour accorder l'accès aux ressources du projet. Les équipes peuvent considérer les projets comme des autorités de certification utilisées pour distribuer ces certificats éphémères. Le niveau de base des certificats contient :
  • Le projet Advanced Server Access pour lequel le certificat a été distribué
  • Le nom d'utilisateur devant être utilisé sur le serveur de l'utilisateur de l'Advanced Access Server auquel le certificat a été délivré
  • L'heure à laquelle le certificat expirera
Utilisateurs Un utilisateur est une personne qui appartient à une équipe et qui s'authentifie avec le fournisseur d'identité de cette équipe. Advanced Server Access définit les permissions des utilisateurs en fonction de leurs appartenances à des groupes.

Les utilisateurs autorisent les clients à les ajouter à leur répertoire, afin qu'ils puissent recevoir leurs identifiants.
Utilisateur de service Les utilisateurs de service sont des comptes spéciaux. Ils ne sont pas liés à une personne physique. Les équipes peuvent utiliser un utilisateur de service pour automatiser des actions en utilisant l'API Advanced Server Access ou pour accorder l'accès à certaines procédures sur la plateforme Advanced Server Access. Consulter la page dédiée aux Utilisateurs de service.
Clients Un client Advanced Server Access est un outil en ligne de commande installé sur une station de travail. Dès lors qu'un utilisateur installe et ajoute le client dans un projet Advanced Server Access, le client permet aux ressources inscrites dans le même projet d'accéder au serveur. Consulter la page dédiée aux Clients de l'Advanced Server Access.
Serveurs L'agent du serveur Advanced Server Access contrôle l'accès au SSH (Secure Shell) et au RDP (Remote Dekstop Protocol) pour les serveurs à distance inscrits dans un projet Advanced Server Access.

Un serveur est uniquement inscrit dans un projet unique. Les équipes peuvent inscrire des serveurs aux projets de façon automatique avec un compte cloud associé ou de façon manuelle à l'aide d'un jeton d'accès. Voir la page dédiée aux Agents du serveur Advanced Server Access .
Comptes des utilisateurs du serveur L'agent du serveur Advanced Server Access gère les comptes des utilisateurs sur les serveurs Windows et Linux.

Si un utilisateur est désactivé dans Okta, l'agent du serveur supprime tous les comptes liés sur le serveur, afin d'empêcher tout accès indésiré.
Droits de l'utilisateur Les droits de l'utilisateur peuvent autoriser les utilisateurs non-administrateurs à utiliser des commandes sudo sans leur accorder de permissions de super utilisateur.

Les équipes peuvent créer un système de couches de permissions basées sur les groupes auxquels un utilisateur appartient.
Attributs Les attributs définissent les différentes caractéristiques des utilisateurs et des groupes. Ils peuvent inclure les noms d'utilisateur Unix ou Windows, les identifiants uniques (UID) ou les identifiants de groupe (GID).

Les équipes peuvent générer des attributs en fonction de paramètres prédéterminés. Les administrateurs contrôlent totalement les attributs des utilisateurs et des groupes de leur équipe. Ils peuvent les modifier pour éviter ou résoudre tout conflit d'attribut dans les déploiements existants.

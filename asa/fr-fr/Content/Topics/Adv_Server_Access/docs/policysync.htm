PolicySync : contrôle d'accès basé sur les attributs.

Cette fonctionnalité est disponible en accès anticipé. Pour l'activer, contactez le Support Okta.

Les équipes Advanced Server Access affectent souvent des ressources à des projets en les regroupant par produit ou par équipe. Par exemple, vous pouvez disposer de projets distincts pour vos équipes chargées de la comptabilité, du marketing et de l'ingénierie. La méthode standard pour accorder l'accès à un serveur Advanced Server Access aux utilisateurs est la suivante : inscription du serveur dans le projet, affectation des utilisateurs à un groupe puis ajout du groupe au projet. Cette méthode accorde l'accès à tous les serveurs qui sont inscrits dans le projet, et ce, pour tous les membres du groupe.

Il est souvent nécessaire d'accorder aux utilisateurs l'accès à un sous-ensemble de serveurs dans un projet. Par exemple, vous pourriez disposer d'une équipe d'administrateurs de base de données (DBA) ayant besoin d'accéder à tous les serveurs de base de données, quel que soit le projet dans lequel le serveur est inscrit. Toutefois, la création d'un projet distinct contenant uniquement les serveurs de base de données briserait le modèle de regroupement des ressources par équipes.

Okta PolicySync permet aux administrateurs Advanced Server Access d'appliquer des contrôles d'accès basés sur les attributs (ABAC) aux groupes d'utilisateurs. Ces contrôles peuvent être utilisés pour accorder un accès aux ressources en fonction de leurs différents rôles. Pour cela, des étiquettes doivent être appliquées aux serveurs et des sélecteurs doivent être appliqués aux groupes, de la même manière que les clients utilisent des sélecteurs d'étiquettes pour identifier les objets dans Kubernetes.

Concepts liés à PolicySync

Trois concepts principaux caractérisent l'utilisation de PolicySync : le fichier des commettants autorisés, les étiquettes et les sélecteurs. La section suivante vous offre une description de chaque concept.

Fichier des commettants autorisés

Okta PolicySync utilise l'option de configuration AuthorizedPrincipalsFile du démon OpenSSH (sshd). Pour en savoir plus sur AuthorizedPrincipalsFile, consultez la section sshd_config.

Lorsque PolicySync est activée, l'agent du serveur Advanced Server Access crée le fichier .asa_authorized_principals et le place dans le répertoire personnel de chaque utilisateur, sur chacun des serveurs auxquels il a accès. Vous pouvez modifier cet emplacement en définissant une option AuthorizedPrincipalsFile dans le fichier de configuration sftd.yaml de votre agent du serveur Advanced Server Access. Pour configurer l'emplacement, vous devez utiliser l'un des jetons utilisateur pris en charge :

  • %h : s'étend au chemin d'accès du répertoire personnel de l'utilisateur
  • %u : s'étend au nom d'utilisateur de l'utilisateur.
  • %U : s'étend à l'identifiant unique (UID) de l'utilisateur.

Par exemple, si vous voulez configurer Advanced Server Access pour qu'il crée un fichier des commettants autorisés dans le répertoire personnel de l'utilisateur, vous pouvez ajouter ce qui suit au fichier de configuration du serveur (sftd.yaml) :

AuthorizedPrincipalsFile: "%h/.asa_authorized_principals"

L'agent du serveur Advanced Server Access met également à jour le fichier sshd_config du serveur avec l'instruction AuthorizedPrincipalsFile définie dans le fichier sftd.yaml.

Bien que l'agent du serveur Advanced Server Acces, sftd, modifie le fichier sshd_config, nous vous recommandons vivement de ne pas le modifier vous-même, que ce soit de manière manuelle ou via l'automatisation. La modification du contenu du fichier sshd_config peut entraîner des configurations non prises en charge ou incorrectes.

Étiquettes

Les étiquettes sont appliquées à des serveurs individuels et permettent aux équipes Advanced Server Access de catégoriser et de trier les accès aux serveurs. Les étiquettes sont stockées au format key:value. Ce dernier permet aux équipes d'être plus flexibles. Par exemple, un serveur peut porter deux étiquettes, env:prod et region:US.

Comme les étiquettes peuvent provenir de sources multiples, Advanced Server Access ajoute automatiquement un préfixe qui permet d'identifier la source de l'étiquette. Par exemple, si l'étiquette env:prod provient de l'API Advanced Server Access, le préfixe api sera ajouté comme suit : api.env:prod. Cette méthode permet d'éviter les conflits dans le cas où la même étiquette a été ajoutée à partir de plusieurs sources (par exemple, si une étiquette a été spécifiée à la fois par l'API et par un fichier de configuration d'agent du serveur Advanced Server Access.

Sélecteurs de serveurs

Les sélecteurs de serveurs sont appliqués aux groupes et contrôlent le nombre de serveurs disponibles pour les membres du groupe. Les sélecteurs de serveurs se caractérisent comme suit :

  • Les utilisateurs d'un groupe dans lequel des sélecteurs sont appliqués ont uniquement accès aux serveurs qui répondent à toutes les exigences de ce sélecteur.
  • Les utilisateurs qui appartiennent à plusieurs groupes ont accès aux serveurs qui appartiennent à l'union des sélecteurs de ces groupes.
  • Les clés des sélecteurs peuvent optionnellement spécifier la source de l'étiquette.
    • Si la clé contient la source, seuls les serveurs qui contiennent l'étiquette de cette source correspondront. Par exemple, si vous utilisez le sélecteur sftd.role=db, seuls les serveurs dont le fichier de configuration sftd.yaml contient l'étiquette role: db correspondront.
    • Si la source n'est pas spécifiée, tout serveur qui contient l'étiquette spécifiée correspondra. Par exemple, si vous utilisez le sélecteur role=db, tout serveur qui contient l'étiquette aws.role: db ou sftd.role: db correspondra.

Nous vous invitons à bien réfléchir à la manière dont vous configurerez vos étiquettes et vos sélecteurs. Vous devez vous assurer que les utilisateurs ont accès aux bastions de leurs projets pour faire en sorte qu'ils puissent se connecter aux serveurs.

Utilisation des étiquettes et des sélecteurs pour les accès granulaires au serveur

Prérequis

Pour utiliser des étiquettes et des sélecteurs dans un projet Advanced Serveur Access, il convient de vérifier les éléments suivants :

  • L'indicateur de fonction PolicySync doit être activé pour votre équipe Advanced Server Access. Pour l'activer, contactez le Support Okta.

  • Les serveurs du projet doivent exécuter l'agent du serveur Advanced Server Access version 1.52.1 ou ultérieure.

Ajout d'une étiquette à un serveur

Les équipes peuvent ajouter des étiquettes depuis le tableau de bord Advanced Server Access. Elles peuvent aussi les ajouter directement dans le fichier de configuration de l'agent du serveur (sftd.yaml). Consulter la section Étiquettes PolicySync.

  1. Ouvrez le tableau de bord Advanced Server Access.
  2. Cliquez sur Projets et ouvrez un projet.
  3. Accédez à l'onglet Serveurs.
  4. Cliquez sur l'engrenage qui se trouve à côté d'un serveur et sélectionnez Edit Labels (Modifier les étiquettes).
  5. Saisissez une ou plusieurs étiquette(s) dans le champ Add or Remove Labels (Ajouter ou supprimer des étiquettes), et appuyez sur la touche Entrée de votre clavier après avoir saisi chaque étiquette.
    Remarque : les étiquettes doivent respecter le format key:value.
  6. Cliquez sur Soumettre pour enregistrer les étiquettes.

Ajout d'un sélecteur au groupe d'un projet

Les équipes peuvent assigner des sélecteurs au niveau du groupe d'un projet. Pour assigner un sélecteur :

  1. Ouvrez le tableau de bord Advanced Server Access.
  2. Cliquez sur Projets et ouvrez un projet.
  3. Accédez à l'onglet Groupes.
  4. Cliquez sur l'engrenage qui se trouve à côté d'un serveur et sélectionnez Modifier.
  5. Sous Accès aux serveurs, sélectionnez Specific Servers(serveurs spécifiques).
  6. Saisissez une ou plusieurs étiquette(s) dans le champ Server Selector Tags (balises de sélecteurs de serveurs) et appuyez sur la touche Entrée de votre clavier après avoir saisi chaque étiquette.
    Remarque : les étiquettes doivent respecter le format key:value.
  7. Cliquez sur Update Group (mettre à jour le groupe) pour enregistrer les sélecteurs.

Client Advanced Server Access

Version : 1.61.0

Date de lancement : 15 juin 2022

Advanced Server Access n'est plus compatible avec les systèmes d'exploitation suivants :

  • Ubuntu 12.04
  • Red Hat Enterprise Linux 6

Pour obtenir la liste des options prises en charge, consultez la section Systèmes d'exploitation pris en charge.

Améliorations

  • Le client Advanced Server Access peut exécuter automatiquement des commandes bash et zsh.

Précédentes versions

Version : 1.60.0

Date de lancement : 4 mai 2022

Dès le début du mois de juin 2022, Advanced Server Access ne sera plus compatible avec les systèmes d'exploitation suivants :

  • Ubuntu 12.04
  • Red Hat Enterprise Linux 6

Si vous utilisez l'un des systèmes d'exploitation mentionnés, Okta vous recommande de passer à un système d'exploitation compatible d'ici juin 2022. Consultez la section Systèmes d'exploitation pris en charge.

Améliorations

  • La commande sft ssh peut inclure la variable d'environnement SFT_ALLOW_INSECURE_USERNAMES=1 afin d'autoriser temporairement les connexions ssh avec des noms d'utilisateur contenant des caractères spéciaux. Consultez la section Utilisation du client Advanced Server Access.
  • Le client est désormais compatible avec l'algorithme d'échange de clés du Groupe Diffie Hellman SHA256 Kex.
  • Diverses modifications ont été apportées à la jointure AD. Consultez la section Jointure AD.
    • Lorsqu'un(e) utilisateur(trice) disposant de plusieurs comptes AD se connecte à un appareil AD découvert, il ou elle peut choisir le compte à utiliser.
    • Les utilisateurs(trices) peuvent se connecter aux appareils AD sans saisir de mot de passe. Consultez la section Certificats sans mot de passe.
    • Les utilisateurs(trices) peuvent se connecter aux appareils AD depuis l'interface utilisateur graphique sur macOS et Windows.
  • Les utilisateurs(trices) peuvent configurer l'option client.timeout_seconds afin de définir un délai maximal d'attente de réponse des serveurs. Consultez la section Configurer le client.
  • Les clients non inscrits affichent désormais des messages d'erreur plus détaillés.

Correctifs

  • La spécification d'un nom d'utilisateur pour les connexions SSH renvoyait une erreur.
  • Le lien vers la documentation Advanced Server Access était incorrect sur le client macOS.
  • Les clients n'initiaient pas de nouveau les connexions liées à certaines opérations si l'équipe avait déjà dépassé les limites de concurrence ou du taux d'API de l'Advanced Server Access.
  • La variable d'environnement SFT_Hook_Events pour les hooks de cycle de vie était indisponible pour les scripts.

Version : 1.51.3

Date de lancement : 23 mars 2021

Améliorations

  • L'agent de serveur valide désormais les modifications apportées au fichier sshd_config. Cela empêche la perturbation de la connectivité pouvant résulter d'un fichier de configuration endommagé.
  • L'agent de serveur crée désormais une sauvegarde du fichier sshd_config avant d'y apporter des changements. Cela permet de s'assurer qu'une configuration qui fonctionne peut être restaurée en cas de fichier de configuration invalide.
  • Le nombre de threads pour les appareils disposant d'un grand nombre de processeurs est désormais réduit. Cela permet d'améliorer les performances respectives de chaque processeur.

Correctifs

  • Lorsque les utilisateurs à la demande essayaient de se connecter à un serveur Windows avec RDP, la première tentative pouvait parfois échouer.

Version : 1.52.2

Date de lancement : 29 avril 2021

Améliorations

  • Les passerelles signalent désormais le nombre de connexions actives. Cela permet aux administrateurs d'avoir une visibilité sur la charge de connexions de la passerelle en temps réel.

  • L'algorithme de validation de l'intégrité du journal en ce qui concerne les journaux de session SSH a été amélioré. Cela permet de garantir l'authenticité des journaux de session SSH.

  • Une passerelle ne démarrera plus si le répertoire temporaire dédié au stockage de journaux ne se situe pas sur un appareil valide. Cela permet de détecter un répertoire de journaux non valide dès le lancement et non plus durant l'exécution.

  • Le répertoire temporaire dédié au stockage de journaux de passerelle peut désormais être spécifié via l'option de configuration SessionLogTempStorageDirectory. Cela permet de configurer le répertoire temporaire, mais également de faire en sorte que ce dernier ainsi que le répertoire de stockage puissent gérer des volumes différents.

Correctifs

  • Les métadonnées de sessions SSH n'étaient pas correctement enregistrées.

Cette modification permet aux administrateurs de déterminer si les fichiers de passerelle sont stockés en local ou de façon externe, et de connaître le volume d'espace utilisé.

Fonctionnalité

  • Le paramètre relatif à l'algorithme de signature de certificat permet l'utilisation par les projets de l'algorithme ssh-ed25519, plus sécurisé, pour les certificats d'authentification de serveurs.
